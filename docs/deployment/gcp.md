# AIGO-X GRC — GCP Deployment Runbook

## Architecture

```
Internet
    │
    ▼
Cloud Load Balancing (global, managed TLS cert)
    │  /api/* → gateway Service (NodePort)
    │  /*      → web Service (NodePort)
    ▼
GKE Autopilot Cluster (private nodes, Workload Identity)
    └── AIGO-X Helm release (aigo-x namespace)
         ├── gateway (8080), auth (8001), risk (8002) …
         └── web (80)
              │
              ├── Cloud SQL PostgreSQL 16
              │    (Regional HA, private IP via Service Networking)
              ├── Memorystore Redis 7 (Standard HA, AUTH, TLS)
              └── Artifact Registry (per-service repos, 12 total)
```

Secrets live in **GCP Secret Manager** and are accessed by pods via **Workload Identity** — the GCP Service Account is bound to the Kubernetes Service Account, giving pods token-based access to secrets without static credentials.

## Step 1 — Authenticate to GCP

```bash
# Interactive (developer workstation)
gcloud auth login
gcloud auth application-default login
gcloud config set project YOUR_PROJECT_ID

# CI/CD (service account key)
export GOOGLE_APPLICATION_CREDENTIALS=/path/to/sa-key.json

# CI/CD (Workload Identity Federation — recommended)
# Configure via: https://cloud.google.com/iam/docs/workload-identity-federation
```

## Step 2 — Enable required APIs

```bash
gcloud services enable \
  container.googleapis.com \
  sqladmin.googleapis.com \
  redis.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com \
  servicenetworking.googleapis.com \
  compute.googleapis.com \
  --project "$GCP_PROJECT_ID"
```

Terraform also enables these automatically on first apply.

## Step 3 — Configure deployment variables

```bash
cp gcp/.env.example gcp/.env
# Edit gcp/.env: set GCP_PROJECT_ID, GCP_REGION, TF_VAR_* etc.
source gcp/.env
```

## Step 4 — Provision infrastructure

```bash
# Plan (no changes made)
GCP_PROJECT_ID=my-project ./gcp/deploy.sh \
  --env prod --project my-project --version 1.0.0 --plan

# Apply (provisions VPC, GKE, Cloud SQL, Memorystore, Artifact Registry, Secret Manager)
GCP_PROJECT_ID=my-project ./gcp/deploy.sh \
  --env prod --project my-project --version 1.0.0
```

Expected duration: **25–40 minutes** (GKE Autopilot + Cloud SQL provisioning).

## Step 5 — Update secrets in Secret Manager

After first deploy, replace CHANGEME placeholder values:

```bash
PROJECT="my-gcp-project"
ENV="prod"
PREFIX="aigo-x-$ENV"

# JWT Secret
echo -n "$(openssl rand -hex 32)" | \
  gcloud secrets versions add "$PREFIX-jwt-secret" --data-file=- --project="$PROJECT"

# Token encryption key
echo -n "$(openssl rand -hex 16)" | \
  gcloud secrets versions add "$PREFIX-token-encryption-key" --data-file=- --project="$PROJECT"

# OpenAI API key
echo -n "$OPENAI_API_KEY" | \
  gcloud secrets versions add "$PREFIX-openai-api-key" --data-file=- --project="$PROJECT"
```

The database password and Redis auth token are auto-generated by Terraform and already stored in Secret Manager. Only update them if you need to rotate.

---

## Step 6 — Configure Workload Identity binding

After Terraform completes and the GKE cluster is ready:

```bash
# Get the GCP Service Account email from Terraform
WI_SA="$(cd gcp/terraform && terraform output -raw workload_identity_sa)"

# Bind it to the Kubernetes Service Account
kubectl create serviceaccount aigo-x-workload-sa -n aigo-x --dry-run=client -o yaml \
  | kubectl annotate --overwrite -f - \
    iam.gke.io/gcp-service-account="$WI_SA" \
  | kubectl apply -f -

# Verify binding
gcloud iam service-accounts get-iam-policy "$WI_SA" --project="$PROJECT"
```

---

## Step 7 — Apply GKE Gateway config

The GKE Gateway and HTTPRoute resources need to be applied after the cluster is ready:

```bash
# The gateway-config.yaml is generated by Terraform
kubectl apply -f gcp/terraform/modules/load-balancing/gateway-config.yaml
```

Wait for the load balancer IP to be assigned:

```bash
kubectl get gateway aigo-x-gateway -n aigo-x -w
```

---

## Step 8 — Subsequent deployments (image update only)

```bash
GKE_CLUSTER_NAME=aigo-x-prod-gke \
GCP_PROJECT_ID=my-project \
IMAGE_TAG=1.2.3 ./gcp/deploy.sh \
  --env prod --project my-project --version 1.2.3 --helm-only
```

## DNS Setup

1. Get the load balancer IP:
   ```bash
   cd gcp/terraform && terraform output load_balancer_ip
   ```
2. Create an A record in your DNS provider:
   ```
   grc.example.com → <LB_IP>
   ```
3. TLS is handled automatically by the Google Managed Certificate (provisioned by Terraform). It activates once DNS propagates (typically 15–60 minutes).

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Pods in `Pending` | Autopilot provisioning nodes | Wait 2–3 min; Autopilot scales on demand |
| `PERMISSION_DENIED` on Secret Manager | Workload Identity not bound | Run Step 6 again |
| Cloud SQL connection refused | Private Services Access not peered | Re-run `terraform apply` |
| Load Balancer shows no healthy backends | Gateway/HTTPRoute not applied | Apply `gateway-config.yaml` |
| Managed cert `PROVISIONING` | DNS not yet propagated | Wait; verify A record |
